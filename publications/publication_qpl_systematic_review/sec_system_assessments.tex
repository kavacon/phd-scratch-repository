\section{System assessments}\label{sec:system_assessments}
The evaluation method defined in Section~\ref{sec:evaluation_instrument} is applied to the eleven primary systems selected in Sections~\ref{sec:taxonomy} and~\ref{sec:methodology}.
Each system is assessed by analysing the code of a representative task suite (T1--T4) and recording condition outcomes (C1--C9).
The code samples are available in Appendix D~\cite{kavadias2026qplSurveySupplementary}.
Per-system evidence summaries are in Subsection~\ref{subsec:system_assessments:per_system}.
Comparative analysis is in Subsection~\ref{subsec:system_assessments:comparative}.

Tower is excluded from the formal assessment; Subsection~\ref{subsec:system_assessments:tower_aside} includes them as labelled reference points.
Findings are reported here; interpretation of RQ1 and RQ2 is in Section~\ref{sec:discussion}.

Each system receives a brief introduction, a task-by-condition outcome matrix, an aggregated condition-profile summary and an evaluation analysis paragraph.

\paragraph{Evaluation analysis}
The recorded outcomes place OpenQASM~3.0 at the Assembly tier.
The four discriminator conditions C2, C3, C6 and C9 all record F, placing the system below the Framework minimum bar defined in Subsection~\ref{subsec:evaluation_instrument:gating}.
The authoring surface across T1--T4 is a gate-level circuit model: every task is a linear sequence of named gate applications on explicit, indexed \texttt{qubit} registers, terminated in T1--T3 by an explicit \texttt{measure} step to declared \texttt{bit} variables.
T4 reduces to the form $\texttt{qubit[2] q; h q[0]; cx q[0], q[1];}$ and the language offers no value-level alternative, so C1, C2, C3 and C5 record sustained F.

Uncomputation is the programmer's responsibility, not a language service.

Two conditions vary across the suite and aggregate worst-case to F.
C9 records C in T1 and T3, where the user-defined \texttt{gate U\_f\_bal} and the modifier expression \texttt{ctrl @ pow(2**k) @ \ldots inv @} carry abstraction, and F in T2 and T4, which inline their gate sequences; abstraction is available but not required.
C8 records C in T1--T3, where interference reads from gate idiom and source comments, and F in T4, where entanglement reduces to a literal \texttt{cx} application; the language admits gate-level descriptions of these phenomena without providing constructs that name them.

C7 records P sustained across T1--T4.
The representative suite uses only \texttt{include "stdgates.inc"} and named gates; basis-set selection, topology, timing constructs \texttt{delay}, \texttt{duration} and \texttt{box}, and pulse-level calibration \texttt{defcal} and OpenPulse are v3 features that are not exercised.
This deviates from the Assembly row of Table~\ref{tab:evaluation_instrument:expected_profile}, which expects F\@.
The canonical examples satisfy the Assembly-tier criteria without engaging v3's hardware-control extensions.

\paragraph{Evaluation analysis}
The recorded outcomes place Qiskit at the Framework tier.
The artefact model under the C2 lens is a host-embedded Python DSL: every task constructs a $\texttt{QuantumCircuit(\ldots)}$ object and mutates it through ordinary method calls, with explicit qubit and classical register identities throughout ($\texttt{qr[0]}$, $\texttt{ClassicalRegister}$, $\texttt{measure\_all()}$).
T4 reduces to the minimal exhibit $\texttt{QuantumCircuit(2); circuit.h(0); circuit.cx(0, 1)}$; the language offers no value-level alternative to the circuit object.
Superposition is a Hadamard call rather than a value, reversal and measurement are hand-written as ordinary code, and the classical bridge is the programmer's responsibility.

The three algorithmic tasks exercise Qiskit's framework affordances: Deutsch--Jozsa composes an oracle via $\texttt{qc.compose(\ldots)}$, Grover powers a library $\texttt{grover\_operator(\ldots)}$, and quantum phase estimation appends $\texttt{QFT(3, inverse=True)}$; each remains followable as a gate recipe with amplitude amplification and phase reasoning left to commentary.
The Bell preparation does neither: a two-gate snippet has nothing to compose and entanglement is the task's goal rather than an effect to reason about.
The framework affordances are exercised when the algorithm calls for them, with circuit-level composition as the core artefact approach rather than value-level functional composition.

The Framework row of Table~\ref{tab:evaluation_instrument:expected_profile} expects C on C5, whereas Qiskit records F\@.
Across the task suite, quantum and classical data remain separated by an explicit register and measurement interface: circuits are provisioned with classical storage and populated with $\texttt{measure(\ldots)}$ statements that the programmer must place and align with qubit identities.
This extracts outcomes but does not lift them into ordinary host values or abstract the measurement boundary.
The interface is therefore manual wiring, not a language service, keeping C5 at F\@.

T1 packages the Deutsch--Jozsa core as a conjugation pattern using $\texttt{within\_apply}$ around $\texttt{hadamard\_transform(x)}$ and $\texttt{phase\_oracle(\ldots)}$, while T2 expresses iteration through $\texttt{power(r,\{...\})}$ and delegates the Grover iterate structure to a library combinator $\texttt{grover\_operator(\ldots)}$.
The observation boundary is present even when it is not written as a raw measurement primitive, for example, via $\texttt{Output[...]}$ parameters and explicit $\texttt{measure(count)}$ in the constructed QPE listing.
By contrast, the phase-estimation task forces a gate-level decomposition and phase-parameter management, including a handwritten two-qubit inverse QFT and controlled phase powers, which drives failures in uncomputation and phase operations.

Relative to the \textit{High-level} expected profile in Table~\ref{tab:evaluation_instrument:expected_profile}, the primary deviations are C1, C4 and C8.
C1 is F because superposition is still introduced through explicit Hadamard applications on addressed elements, C4 is F because the representative phase-estimation construction requires manual inverse-QFT decomposition as ordinary code.
C8 is F because both phase estimation and Bell-state preparation require explicit phase and entanglement understanding for correctness rather than remaining purely recipe-level.

\subsection{Tower reference aside}\label{subsec:system_assessments:tower_aside}
\textit{The following is an informal aside.
Tower was excluded from primary assessment; its outcomes do not contribute to the comparative analysis or the conclusions in Section~\ref{sec:discussion}.
They are included as labelled reference points for contrasting system approaches and evidence limitations.}

Tower is retained as a labelled contrast because it develops a programming model for pointer-based quantum data structures and dynamic memory allocation, and it treats the constraints required for correctness under superposition as first-class language concerns.\cite{yuan2025foundationalabstractions}
The Tower model targets programs whose state may be a superposition of configurations, so a single operation must act uniformly across that superposition.\cite{yuan2025foundationalabstractions}
The formal development characterises three constraints that shape its data model: operations are required to be reversible, data structures are required to be history independent and each operation is required to execute in bounded time determined by classical parameters.\cite{yuan2025foundationalabstractions}
Tower enforces reversibility through an explicit assignment and unassignment discipline, and it introduces a memory-allocation mechanism intended to preserve history independence under allocation and deallocation.\cite{yuan2025foundationalabstractions}

Tower is excluded from the task-based assessment and it does not contribute to the tier distribution in Subsection~\ref{subsec:system_assessments:summary} or the comparative analysis in Subsection~\ref{subsec:system_assessments:comparative}.
Under the pre-specified inclusion criteria it fails the 24-month maintenance window and it does not provide task-aligned public documentation sufficient to implement the representative task suite T1--T4 in a comparable way.
Tower’s distinctive contribution is expressed in its data model and reversible programming discipline rather than through implementations of the selected algorithmic tasks.
Figure~\ref{fig:pub_qpl_review:system_assessments:tower_list} provides a representative example of using lists in Tower.
